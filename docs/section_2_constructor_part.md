# 2 КОНСТРУКТОРСКАЯ ЧАСТЬ

На основе проведенного анализа требований и существующих решений в области платформ для торговли цифровыми фотографиями, следующим этапом разработки является проектирование архитектуры веб-приложения "ФотоТочка". Данный раздел описывает детали реализации системы, включая структуру клиентской и серверной частей, организацию взаимодействия между компонентами и механизмы обработки данных.

## 2.1 Разработка архитектуры веб-приложения

Архитектура веб-приложения "ФотоТочка" основана на принципах клиент-серверного взаимодействия, что обеспечивает разделение ответственности между пользовательским интерфейсом и бизнес-логикой приложения. Выбор данной архитектурной модели обусловлен необходимостью создания масштабируемой, производительной и удобной в поддержке системы, способной эффективно обрабатывать большое количество запросов от пользователей и обеспечивать быстрый доступ к медиафайлам.

Клиентская часть приложения реализована как одностраничное приложение (Single Page Application, SPA), которое работает непосредственно в браузере пользователя и обеспечивает динамическое обновление контента без необходимости перезагрузки страницы. Серверная часть представляет собой программное обеспечение, функционирующее на удаленном сервере и отвечающее за обработку бизнес-логики, работу с базой данных и предоставление данных клиентскому приложению через программный интерфейс.

Взаимодействие между клиентской и серверной частями осуществляется посредством REST API (Representational State Transfer Application Programming Interface), который обеспечивает стандартизированный способ обмена данными в формате JSON (JavaScript Object Notation). Такой подход позволяет клиентской и серверной частям разрабатываться и обновляться независимо друг от друга, что существенно упрощает процесс разработки и поддержки приложения.

Преимущества выбранной архитектуры проявляются в возможности горизонтального масштабирования серверной части при увеличении нагрузки, использовании кеширования на стороне клиента для снижения количества запросов к серверу, а также в улучшенной производительности за счет асинхронной загрузки данных и оптимизации передачи информации. Разделение на независимые слои также способствует повышению безопасности приложения, поскольку критически важные операции выполняются на защищенном сервере, а не в браузере пользователя.

## 2.2 Архитектура клиентской части (Frontend)

### 2.2.1 Структура компонентов Vue.js

Клиентская часть веб-приложения построена с использованием компонентного подхода, который является фундаментальным принципом современной разработки пользовательских интерфейсов. Компонентный подход предполагает разделение интерфейса на независимые, переиспользуемые элементы, каждый из которых инкапсулирует свою логику, структуру и стили. Такая организация кода обеспечивает модульность, упрощает тестирование компонентов и способствует созданию единообразного пользовательского интерфейса.

В контексте приложения "ФотоТочка" компоненты организованы в иерархическую структуру, где базовые компоненты используются для построения более сложных составных компонентов, формирующих страницы приложения. Иерархия включает атомарные элементы (кнопки, иконки), составные компоненты (карточки, формы) и компоненты страниц, объединяющие множественные элементы в полноценные экраны.

Переиспользование компонентов является ключевым аспектом эффективной разработки интерфейса. Компонент PhotoCard, предназначенный для отображения информации о фотографии, может использоваться в различных контекстах: на главной странице для демонстрации популярных фотографий, в каталоге для представления результатов поиска, в личном кабинете фотографа для отображения портфолио. Адаптивность компонента обеспечивается за счет системы пропсов (props), которые позволяют передавать различные данные и настраивать внешний вид компонента в зависимости от контекста использования.

Основные компоненты приложения формируют комплексную систему пользовательского интерфейса, где каждый элемент выполняет строго определенную функцию. Компонент HeaderNav обеспечивает навигацию по приложению и отображение контекстного меню в зависимости от статуса авторизации пользователя. Компонент PhotoCard предназначен для представления отдельной фотографии с основной информацией и служит строительным блоком для различных частей интерфейса. Компонент PhotoGrid организует отображение множества фотографий в сеточной структуре с автоматической адаптацией количества колонок под размер экрана устройства. Компонент SearchFilters предоставляет интерфейс для фильтрации результатов поиска по различным критериям, позволяя пользователям уточнять запросы. Компонент LicenseSelector реализует функциональность выбора типа лицензии при оформлении покупки фотографии, предоставляя пользователю информацию о доступных вариантах и их условиях. Компонент AuthorShowcase демонстрирует информацию об авторе фотографии, включая портфолио и контактные данные. Специализированные компоненты для административной панели и личного кабинета фотографа обеспечивают функциональность управления контентом и настройками платформы. Каждый компонент спроектирован с учетом принципов единственной ответственности, что означает сосредоточение на решении одной конкретной задачи с максимальной эффективностью и возможностью независимого тестирования и модификации.

Пример реализации компонента PhotoCard демонстрирует практическое применение принципов компонентной архитектуры. Компонент использует систему пропсов для настройки поведения и внешнего вида, что обеспечивает его переиспользование в различных контекстах приложения.

```typescript
// Компонент PhotoCard.vue - пример использования пропсов и эмитов
<script setup lang="ts">
interface Photo {
  id: string
  title: string
  category: string
  imageUrl: string
  status?: 'published' | 'pending' | 'draft'
}

interface Props {
  photo: Photo
  showActions?: boolean
  showStats?: boolean
  variant?: 'default' | 'compact' | 'category'
}

withDefaults(defineProps<Props>(), {
  showActions: false,
  showStats: false,
  variant: 'default',
})

const emit = defineEmits<{
  edit: [id: string]
  delete: [id: string]
}>()
</script>

<template>
  <div :class="['photo-card', `photo-card--${variant}`]">
    <div class="photo-card__image">
      <img :src="photo.imageUrl" :alt="photo.title" />
      <div v-if="showActions" class="photo-card__overlay">
        <button @click="emit('edit', photo.id)">Редактировать</button>
        <button @click="emit('delete', photo.id)">Удалить</button>
      </div>
    </div>
    <div class="photo-card__body">
      <h3>{{ photo.title }}</h3>
      <p>{{ photo.category }}</p>
    </div>
  </div>
</template>
```

Данный пример иллюстрирует ключевые особенности реализации компонента: использование TypeScript интерфейсов для типизации пропсов, систему значений по умолчанию через `withDefaults`, механизм эмитов для взаимодействия с родительским компонентом, а также условную отрисовку элементов через директиву `v-if`. Варианты компонента (`variant`) позволяют адаптировать его внешний вид под различные контексты использования, демонстрируя практическое применение принципа переиспользования кода.

### 2.2.2 Маршрутизация и навигация

Навигация по приложению реализована с использованием Vue Router, который является официальной библиотекой маршрутизации для Vue.js и предоставляет возможности для создания многостраничного интерфейса в рамках одностраничного приложения. Маршрутизация обеспечивает соответствие между URL-адресами в браузере и компонентами, которые должны отображаться пользователю, создавая иллюзию работы с традиционным многостраничным веб-сайтом при сохранении преимуществ SPA.

Структура маршрутов приложения организована по принципу разделения доступа, где выделяются три основные категории с различным уровнем защиты. Публичные маршруты доступны всем пользователям без ограничений и включают главную страницу приложения, каталог фотографий и раздел блога. Защищенные маршруты требуют предварительной авторизации и предоставляют доступ к функциональности личного кабинета фотографа, включая управление портфолио, загрузку новых работ и просмотр статистики. Административные маршруты доступны только пользователям с соответствующими правами и обеспечивают доступ к панели управления платформой, модерации контента и управлению системными настройками.

[Рисунок 2.3 — Навигационное меню приложения (компонент HeaderNav)]

На рисунке 2.3 представлен компонент навигации HeaderNav, демонстрирующий реализацию адаптивного меню с поддержкой мобильной версии. Компонент отображает логотип платформы, основные навигационные ссылки, а также кнопки входа и регистрации. При уменьшении размера экрана меню преобразуется в гамбургер-меню с выдвижной панелью, что обеспечивает удобство использования на всех устройствах.

Защита маршрутов реализована через систему метаданных, которые ассоциируются с каждым маршрутом и определяют требования к доступу. Перед переходом на защищенный маршрут система проверяет наличие у пользователя необходимых прав доступа. Если пользователь не авторизован, он перенаправляется на страницу входа с сохранением информации о запрашиваемой странице, что позволяет автоматически перенаправить пользователя после успешной авторизации. Аналогичным образом реализована защита административных маршрутов, где дополнительно проверяется роль пользователя.

```typescript
// router/index.ts - конфигурация маршрутов с защитой доступа
router.beforeEach((to, _from, next) => {
  if (to.meta.requiresAuth && !auth.isAuthenticated()) {
    next({ name: 'login', query: { redirect: to.fullPath } })
  } else if (to.meta.requiresAdmin && !auth.isAdmin()) {
    next({ name: 'home' })
  } else {
    next()
  }
})

const routes = [
  {
    path: '/admin',
    name: 'admin',
    component: () => import('../pages/AdminPage.vue'),
    meta: { requiresAuth: true, requiresAdmin: true },
  },
  {
    path: '/photographer',
    name: 'photographer-dashboard',
    component: () => import('../pages/PhotographerDashboard.vue'),
  },
]
```

Приведенный код демонстрирует реализацию защиты маршрутов через навигационный гард `beforeEach`. Механизм проверяет наличие метаданных `requiresAuth` и `requiresAdmin` в конфигурации маршрута, выполняет проверку авторизации пользователя через утилиту `auth`, и при отсутствии необходимых прав перенаправляет пользователя на соответствующую страницу. Для защищенных маршрутов сохраняется исходный путь запроса в параметрах, что позволяет вернуть пользователя после успешной авторизации.

Обработка несуществующих страниц осуществляется через специальный маршрут-перехватчик, который срабатывает при попытке доступа к несуществующему URL и автоматически перенаправляет пользователя на главную страницу. Такое решение обеспечивает корректную работу приложения даже при вводе неверных адресов и предотвращает отображение ошибок, что улучшает пользовательский опыт.

### 2.2.3 Управление состоянием приложения

Управление состоянием приложения является критически важным аспектом функционирования клиентской части, поскольку от эффективности этой системы зависит отзывчивость интерфейса и согласованность данных в различных компонентах. Vue.js предоставляет встроенную систему реактивности, которая автоматически отслеживает изменения в данных и обновляет соответствующие части интерфейса без необходимости ручного управления обновлениями DOM.

Реактивность Vue.js основана на системе отслеживания зависимостей, где каждое реактивное свойство автоматически обновляет все зависимые компоненты при изменении значения. Локальное состояние компонентов управляется через функции ref или reactive из Composition API для данных, используемых исключительно в рамках конкретного компонента, таких как состояние полей форм, флаги открытия модальных окон или временные значения счетчиков. Вместе с тем, управление данными пользователя, которые должны сохраняться между сессиями, реализовано через специализированную утилиту авторизации, хранящую информацию в механизме localStorage браузера для обеспечения сохранности сессии при перезагрузке страницы. Кеширование данных каталога фотографий оптимизирует производительность приложения, сохраняя загруженные данные в памяти приложения и обновляя их при необходимости, что снижает количество запросов к серверу и ускоряет отклик интерфейса.

## 2.3 Архитектура серверной части (Backend)

### 2.3.1 Структура Django-приложения

Серверная часть веб-приложения построена на основе фреймворка Django, который следует архитектурному паттерну Model-View-Template (MVT), адаптированному для веб-разработки. Django предоставляет модульную структуру организации кода, где каждый модуль, называемый приложением (app), отвечает за определенную функциональную область системы. Такая организация способствует разделению ответственности, упрощает разработку, тестирование и поддержку кода.

Проект "ФотоТочка" организован как совокупность нескольких Django-приложений, каждое из которых фокусируется на определенной функциональной области системы. Приложение users централизует управление пользователями платформы, охватывая процессы регистрации новых участников, авторизации существующих пользователей, управления персональными профилями и распределения прав доступа в зависимости от роли пользователя. Приложение photos концентрируется на работе с фотографиями как основным контентом платформы, обеспечивая функциональность загрузки изображений, их хранения в файловой системе или облачном хранилище, обработки и оптимизации файлов, каталогизации с добавлением метаданных и предоставления структурированных данных о фотографиях через программный интерфейс. Приложение orders реализует бизнес-логику электронной коммерции, включая формирование заказов пользователей, управление корзиной покупок, обработку транзакций и ведение истории приобретений. Приложение categories отвечает за организацию системы категоризации контента, позволяя структурировать фотографии по тематическим группам и создавать иерархическую структуру подкатегорий для удобной навигации и поиска.

Каждое Django-приложение содержит модели данных, представления (views), сериализаторы для API, URL-конфигурацию и административный интерфейс. Модели данных определяют структуру информации, хранящейся в базе данных, и взаимоотношения между различными сущностями. Представления обрабатывают HTTP-запросы, выполняют бизнес-логику и формируют ответы для клиентского приложения. Сериализаторы преобразуют объекты моделей в формат JSON для передачи через API и обратно для сохранения в базе данных.

Использование Django REST Framework расширяет возможности Django для создания RESTful API, предоставляя набор инструментов для сериализации данных, обработки запросов, аутентификации и авторизации, пагинации результатов, фильтрации и поиска. Django REST Framework интегрируется с базовым функционалом Django, дополняя его специализированными возможностями, необходимыми для создания современного API.

### 2.3.2 Модели данных

Модели данных представляют собой абстракцию структуры базы данных и определяют, какая информация хранится в системе и как различные сущности связаны между собой. Модель User расширяет базовую модель пользователя Django дополнительными полями, необходимыми для функционирования платформы, включая роль пользователя (покупатель, продавец, администратор), аватар, биографию, настройки уведомлений. Модель Photo содержит информацию о фотографии: название, описание, файл изображения, категорию, теги, метаданные о разрешении и размере файла, статистику просмотров и загрузок, статус модерации, связь с автором.

Модель Category определяет категории, к которым могут относиться фотографии, и включает название, описание, иконку и иерархическую структуру с возможностью создания подкатегорий. Модель Order представляет информацию о заказах пользователей, включая список приобретенных фотографий, тип лицензии, общую стоимость, статус заказа, дату создания и данные о покупателе. Модель License определяет типы лицензий, доступные для фотографий, их стоимость, ограничения использования и описание условий применения.

Связи между моделями реализованы через механизмы реляционной базы данных, где ForeignKey обеспечивает связь типа "многие-к-одному", что позволяет, например, связывать каждую фотографию с единственным автором, в то время как один автор может создавать неограниченное количество фотографий. Для более сложных связей применяется ManyToMany, который используется для связи "многие-ко-многим", позволяя фотографии относиться одновременно к нескольким категориям, а категории содержать множество различных фотографий. Связи между моделями настроены с использованием каскадного удаления для автоматической очистки зависимых объектов или с защитой от удаления при наличии связанных записей, что обеспечивает целостность данных в системе при выполнении операций удаления объектов.

Поля моделей снабжены индексами для оптимизации производительности запросов к базе данных, что особенно важно при работе с большими объемами информации. Индексы создаются для полей, которые активно используются в условиях фильтрации и сортировки, включая дату создания фотографии, категорию контента и статус модерации, что существенно ускоряет выполнение типичных операций поиска и отбора данных. Дополнительные составные индексы оптимизируют сложные запросы, которые фильтруют данные по нескольким полям одновременно, что характерно для системы поиска и фильтрации в каталоге фотографий, где пользователи могут одновременно задавать критерии по категории, ориентации, разрешению и другим параметрам.

### 2.3.3 REST API и эндпоинты

REST API приложения построен в соответствии с принципами REST архитектуры, которая определяет стандартизированный способ организации взаимодействия между клиентом и сервером через HTTP-протокол. REST архитектура предполагает использование стандартных HTTP-методов для выполнения различных операций над ресурсами, где метод GET применяется для получения данных о существующих ресурсах, POST служит для создания новых ресурсов в системе, методы PUT и PATCH обеспечивают обновление существующих ресурсов с различной степенью полноты изменений, а метод DELETE используется для удаления ресурсов из системы. Каждый эндпоинт API соответствует определенному ресурсу и предоставляет набор операций, которые логически применимы к данному типу сущностей в контексте бизнес-логики приложения.

Основные эндпоинты API организованы по ресурсам, каждый из которых предоставляет набор операций для работы с соответствующими сущностями. Для работы с фотографиями предусмотрены эндпоинты GET /api/photos, позволяющий получать список фотографий с поддержкой пагинации и многоуровневой фильтрации, POST /api/photos для загрузки новых изображений авторами, и GET /api/photos/{id} для получения детальной информации о конкретной фотографии. Работа с пользователями реализована через эндпоинты GET /api/users/me для получения информации о текущем авторизованном пользователе, PUT /api/users/me для обновления данных профиля, и GET /api/users/{id} для просмотра публичной информации о других пользователях. Функциональность заказов обеспечивается эндпоинтами GET /api/orders для просмотра истории заказов текущего пользователя, POST /api/orders для создания нового заказа, и GET /api/orders/{id} для получения детальной информации о конкретном заказе. Сериализация данных выполняется через механизм Django REST Framework Serializers, который определяет структуру ответа API, выполняет преобразование типов данных между представлением в базе данных и JSON-форматом, а также обеспечивает валидацию входящих запросов на соответствие ожидаемой структуре и типам данных.

Аутентификация и авторизация реализованы через систему токенов, где при успешной авторизации пользователь получает JSON Web Token (JWT), который должен передаваться в заголовке Authorization при каждом запросе к защищенным эндпоинтам. Сервер проверяет валидность токена и права доступа пользователя к запрашиваемому ресурсу, возвращая ошибку 401 при отсутствии или невалидности токена и ошибку 403 при недостаточных правах доступа.

## 2.4 Интеграция клиентской и серверной частей

### 2.4.1 Механизм взаимодействия

Взаимодействие между клиентской и серверной частями приложения осуществляется через HTTP-протокол с использованием формата JSON для представления данных. Клиентское приложение формирует HTTP-запросы с указанием метода запроса, URL эндпоинта, заголовков, содержащих информацию об авторизации и формате данных, и тела запроса для методов POST, PUT, PATCH. Сервер обрабатывает запрос, выполняет необходимые операции с базой данных, формирует ответ и отправляет его клиенту в формате JSON.

Асинхронное взаимодействие обеспечивается через механизмы Promise и async/await, что позволяет не блокировать пользовательский интерфейс во время ожидания ответа от сервера, обеспечивая плавную работу приложения даже при задержках сетевого соединения. Обработка ошибок реализована через систему перехвата исключений, которая различает различные типы ошибок, включая проблемы сетевого соединения, ошибки валидации введенных данных, проблемы авторизации и серверные ошибки, применяя для каждого типа соответствующую стратегию обработки с информированием пользователя о характере возникшей проблемы.

### 2.4.2 Безопасность передачи данных

Безопасность передачи данных обеспечивается применением протокола HTTPS с шифрованием TLS, которое предотвращает перехват и модификацию данных при передаче по сети, защищая как персональную информацию пользователей, так и финансовые транзакции. Аутентификация пользователей реализована через механизм JWT-токенов, содержащих закодированную информацию о пользователе и подписанных секретным ключом сервера для предотвращения подделки. Токен передается в заголовке Authorization каждого запроса к защищенным эндпоинтам и имеет ограниченный срок действия для повышения безопасности, при этом механизм refresh-токенов позволяет продлевать сессию без повторного ввода учетных данных. Защита от CSRF-атак реализована через систему токенов, автоматически проверяемых Django для всех запросов, изменяющих состояние системы. Валидация данных выполняется на двух уровнях: на клиенте для обеспечения немедленной обратной связи пользователю при вводе некорректных данных, и на сервере для гарантированного обеспечения безопасности системы, поскольку клиентская валидация может быть обойдена злоумышленниками.

## 2.5 Система хранения и обработки медиафайлов

Хранение загруженных фотографий организовано с использованием файловой системы сервера с возможностью последующего перехода на облачное хранилище для обеспечения масштабируемости и отказоустойчивости. Файлы сохраняются в структурированных директориях, организованных по датам загрузки или идентификаторам авторов, что упрощает управление файлами и обеспечивает равномерное распределение нагрузки на файловую систему. Имена файлов генерируются уникальным образом для предотвращения конфликтов и обеспечения безопасности.

Генерация превью (thumbnails) выполняется автоматически при загрузке фотографии для оптимизации отображения в каталоге и снижения нагрузки на сеть при передаче данных. Превью создаются в нескольких размерах: маленькое для миниатюр в сетке каталога, среднее для детальной страницы фотографии, большое для полноэкранного просмотра. Генерация превью оптимизирует использование дискового пространства и пропускной способности сети, так как загружается только необходимый размер изображения в зависимости от контекста использования.

Оптимизация изображений для веб выполняется через сжатие файлов с сохранением визуального качества. Современные алгоритмы сжатия позволяют существенно уменьшить размер файлов без заметной потери качества, что особенно важно для высокоразрешенных фотографий. Использование формата WebP обеспечивает лучшее сжатие по сравнению с традиционными форматами JPEG и PNG, при этом сохраняя совместимость с современными браузерами. Для браузеров, не поддерживающих WebP, применяется автоматическое определение и предоставление альтернативных форматов.

Ограничения по размеру и формату файлов устанавливаются для предотвращения загрузки некорректных файлов и контроля использования ресурсов сервера. Максимальный размер файла ограничен разумными пределами, которые обеспечивают баланс между качеством изображений и производительностью системы. Поддерживаются стандартные форматы изображений: JPEG для фотографий, PNG для изображений с прозрачностью, WebP для оптимизированных изображений. Валидация формата выполняется как по расширению файла, так и по его содержимому для предотвращения обхода ограничений через переименование файлов.
